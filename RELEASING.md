# Releasing

Releases are intentionally manual while the packages are at `0.1.x`.

## Prerequisites

- Confirm that the MIT license appears in both package tarballs.
- Authenticate as an owner or publisher of the `@phaser-game-engines` npm
  organization. The initial owner is `macardoso95`; verify with
  `npm org ls phaser-game-engines` before publishing.
- Start from a clean checkout on a supported Node version.

## Checklist

1. Move relevant entries from `Unreleased` in `CHANGELOG.md` into a versioned
   section and include migrations for every breaking change.
2. Run `npm run version:packages -- <version>` to align publishable package
   versions, the private root version, internal dependency ranges, and the
   lockfile.
3. Run `npm run check:release`. It is read-only and reports repository-controlled
   blockers; it does not contact npm or publish anything.
4. Run `npm ci` followed by `npm run verify`.
5. Inspect the toolkit and generator tarballs with
   `npm pack --dry-run --workspace <package-name>`.
6. Publish with a prerelease tag and npm provenance until the `1.0` acceptance
   criteria are met. npm provenance must be generated by a supported cloud CI
   runner; do not pass `--provenance` from a local release shell. The initial
   manual publication may establish the package names before trusted publishing
   is configured.
7. Install the published versions in a clean external project before creating a
   signed Git tag and GitHub release containing the changelog section.

Do not publish from a working tree with uncommitted changes. Add publishing
automation after the first manual prerelease validates organization access,
provenance, package visibility, and the registry-install smoke test.
